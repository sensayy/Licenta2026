Accuracy — the simplest metric. Out of all samples, what percentage did the model predict correctly. 99.63% means 4 out of 1088 were wrong.

Balanced Accuracy — same as accuracy but averaged per class first. This matters because your dataset is imbalanced — Simda has only 6 test samples while Kelihos_v3 has 293. Regular accuracy is dominated by the large classes, so a model that ignores Simda entirely can still look good. Balanced accuracy gives each class equal weight regardless of how many samples it has. A score close to regular accuracy means your model performs consistently across all classes, not just the big ones.

Top-2 Accuracy — was the correct class at least in the model's top 2 predictions? Useful for understanding how "close" the wrong predictions are. If Top-2 is 99.9% but accuracy is 99.63%, it means almost every mistake was a near-miss where the right answer was the second choice.

Log Loss — measures the quality of the probability estimates, not just the final prediction. A model that says "I'm 99% sure it's Ramnit" and is right gets a better log loss than one that says "I'm 51% sure." Lower is better. A perfect model would have log loss of 0. This penalizes overconfident wrong predictions very harshly, so it's a good sanity check against the ensemble being poorly calibrated.

Matthews Correlation Coefficient (MCC) — arguably the most honest single-number summary for multiclass classification, especially with imbalanced classes. It takes into account true positives, true negatives, false positives, and false negatives all at once. The scale runs from -1 (perfectly wrong) to 0 (no better than random guessing) to +1 (perfect). A score of 0.99 means the model is extremely strong across all classes simultaneously, not just on the easy majority classes.

AUC-ROC (per class) — stands for Area Under the Receiver Operating Characteristic Curve. For each class it asks: if I vary the confidence threshold from 0 to 1, how well can the model separate "this class" from "everything else"? The curve plots the true positive rate against the false positive rate at every threshold. AUC = 1.0 means perfect separation. AUC = 0.5 means the model is no better than a coin flip for that class. It's computed one-vs-rest here, meaning for each of the 9 classes it treats that class as positive and all others as negative. This is useful because it tells you not just whether predictions are right but how confidently and cleanly the model distinguishes each malware family.

Precision (per class) — of all the samples the model predicted as class X, what fraction actually were class X. High precision means few false alarms.
Recall (per class) — of all the samples that actually are class X, what fraction did the model correctly identify. High recall means few missed detections.
F1-Score — the harmonic mean of precision and recall. Useful when you care about both equally. A class can have high precision but low recall (model is cautious, only predicts it when very sure) or the reverse. F1 captures the balance.

Confidence Threshold / Unknowns — the model outputs a probability for each class. The highest probability across the 9 classes is the confidence score. If that score is below 0.55, the prediction is flagged as "Unknown" rather than committing to a class. This is a safety mechanism — it's better to say "I don't know" than to make a wrong prediction with low confidence. The Unknown count tells you how often the ensemble was genuinely uncertain.

Grad-CAM (Gradient-weighted Class Activation Mapping) — a technique that answers "which pixels in the image influenced this prediction the most?" It works by taking the gradients of the predicted class score with respect to the last convolutional layer's feature maps, then weighting those feature maps accordingly and overlaying them as a heatmap on the original image. Red/warm areas are where the CNN was "looking" when it made its decision. For malware images this is particularly interesting because it can reveal whether the model is focusing on structurally meaningful regions like the file header, the code section, or the data section — or whether it's picking up on something less interpretable.

Top-20 Feature Importance (XGBoost) — XGBoost builds an ensemble of decision trees. Each time a feature is used to split a node in a tree, it contributes to reducing the loss. "Gain" importance sums up how much loss reduction each feature contributes across all trees and all splits. The top 20 features are the ones the model relied on most heavily. In your case these are tabular features extracted from the assembly/byte files — things like n-gram frequencies, opcode counts, or byte value statistics depending on what your feature extraction pipeline produced.
